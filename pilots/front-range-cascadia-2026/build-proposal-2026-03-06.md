# Build Proposal - 2026-03-06

## Title

Boulder / Front Range Commons Packet Sprint

## Purpose

This is the build I would propose for Friday, March 6, 2026.

It is meant to answer the live questions in the Telegram thread without opening another abstract architecture round.

The immediate need is not just "build something cool." The immediate need is to create one concrete artifact that ties together:

- Boulder / Front Range boundary clarity
- indicator-oriented ecological data work
- voice / interview / assembly knowledge capture
- community-facing commons workflows

## The Core Proposal

Build one small but real **commons packet** that includes:

1. a working Boulder / Front Range boundary model
2. one small indicator overlay
3. one interview / assembly / transcript artifact processed into governed knowledge artifacts

The output should be a reusable packet, not just a concept note.

## Working Boundary Choice for Today

Do not try to settle the one true Boulder / Front Range boundary in a single build session.

Use a layered boundary model instead:

### Layer 1: Regional / social frame

- `Front Range`
- `Boulder`
- `Boulder-Denver corridor` where useful

This is the organizing frame for people, organizations, projects, and coalition activity.

### Layer 2: Primary hydrologic frame for the Boulder use case

- `Boulder Creek watershed`
- `St. Vrain watershed`
- `South Platte basin`

This is the frame to use for the first Boulder-oriented environmental data prototype.

### Layer 3: Related upstream / cross-basin dependency frame

- upper Colorado headwaters
- transmountain diversion / water-supply links
- broader Colorado River Basin references where relevant

This should be modeled as related context, not as the only boundary definition for Boulder.

## Important Clarification

The current Front Range node already contains Colorado River Basin watershed data because that was a fast path to getting real hydrologic context into the graph.

That is useful.

But for the **Boulder-specific** use case, I would not treat the full Colorado River Basin as the primary working boundary. I would treat it as a broader context layer around a more local Boulder / St. Vrain / South Platte framing.

## Concrete Outputs for the Session

### 1. Boundary Packet

Create one packet that names and relates:

- Boulder
- Front Range
- Boulder Creek watershed
- St. Vrain watershed
- South Platte basin
- any relevant upstream Colorado water dependency

Key output:

- explicit entities
- explicit relations
- explicit note on which layer is primary for which use case

### 2. Indicator Overlay

Attach one small indicator set to that packet.

Good first candidates:

- snowpack / SNOTEL
- streamflow / gauge data
- one locally meaningful social or stewardship indicator

The goal is not comprehensive coverage. The goal is one usable overlay tied to specific places, questions, and problems.

### 3. Voice-to-Graph Artifact

Take one interview, meeting excerpt, or assembly artifact and run it through a consent-aware extraction path.

Desired outputs:

- `Question`
- `Claim`
- `Practice`
- `Pattern`
- `Protocol`
- `Decision`
- `NextAction`

If some parts should remain local, that is part of the output too. The build should surface the consent boundary, not pretend it does not exist.

This maps directly to the PACTO Evidence Trail — the extraction path creates records at each layer (assembly metadata, voice capture provenance, synthesis artifacts, review decisions) alongside the content-level outputs. The PACTO framework's progressive disclosure model (public → verified → authorized → community only) is already implemented as visibility scopes in the graph's consent membrane.

### 4. Optional Community Workflow Layer

If Benjamin or others want to work the community-facing side, this packet is a real thing for Commons Engine to edit, moderate, and publish against.

That is better than discussing UX in the abstract.
